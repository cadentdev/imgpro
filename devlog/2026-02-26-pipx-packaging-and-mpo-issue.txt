  pipx entry point: Runs imgpro:main() in an isolated venv with its own copy of
  dependencies.

  Key differences:

  1. Python environment вҖ” The wrapper uses the project's .venv, so it shares
  dependencies with your dev environment. pipx creates its own isolated venv.
  2. Code source вҖ” The wrapper runs imgpro.py directly from the repo (live edits
   are immediately reflected). pipx installs a copy, so you'd need to pipx 
  install . --force after changes.
  3. Portability вҖ” The wrapper breaks if you move/delete the repo or the .venv.
  pipx is self-contained.

  For active development on this project, the wrapper is arguably more
  convenient since changes are instant. pipx is better for a stable "installed
  tool" experience. You could also get the best of both worlds with pipx install
   --editable ., which installs in pipx's isolated venv but links back to your
  source so edits are reflected immediately.

вҸә Interesting idea. A few thoughts:

  It adds complexity to the user flow. Right now every command takes one file in
   and produces output. Extracting all MPO frames turns a single input into
  multiple outputs before the user's actual operation вҖ” it's a new preprocessing
   step that doesn't fit neatly into any existing command.

  In practice, MPO frames aren't equal alternatives. iPhone MPO files typically
  contain:
  - Frame 0: the full-resolution primary image
  - Frame 1+: a smaller thumbnail or depth map

  So "select the best frame" usually means "frame 0" by default. Other cameras
  (3D/stereo) may have genuinely different frames, but that's a niche case.

  A middle ground might work better:

  1. Default behavior: Normalize MPO вҶ’ JPEG (use frame 0) вҖ” this is the fix for
  the bug and handles 95% of cases
  2. New feature: Add an info flag or a dedicated subcommand to inspect/extract
  MPO frames when the user actually needs it (e.g., imgpro info photo.jpg 
  --frames to list them, or imgpro convert photo.jpg --format jpeg --all-frames
  to export each)

  This keeps the common path simple while giving power users access to all
  frames. Does that feel right, or do you want to prioritize the multi-frame
  extraction from the start?
